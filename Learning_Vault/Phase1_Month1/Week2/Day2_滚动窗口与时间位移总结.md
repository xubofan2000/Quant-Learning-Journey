# Phase 1 - Week 2 - Day 2: 滚动窗口与时间位移总结

## 1. `.rolling()` = SQL 窗口函数的 Python 化身
你之前在写 SQL 时，如果要算"过去 5 天的平均收盘价"，需要写这样一段语法：
```sql
AVG(close_price) OVER (ORDER BY trade_date ROWS BETWEEN 4 PRECEDING AND CURRENT ROW)
```
而在 Pandas 中，这行复杂的窗口声明被压缩成了极简的一句：
```python
df['close_price'].rolling(window=5).mean()
```
* **`window=5` 的物理含义**：以当前行为锚点，往回数 4 行，加上自己共 5 行数据，送入 `.mean()` 求平均。然后窗口整体向下滑动一格，重复这个过程。
* **前 N-1 行为 NaN 的原因**：第 1 天只有自己 1 个数据点，凑不够 5 个，Pandas 老老实实返回"无法计算"。这不是 Bug，而是严谨——如果强行用不足 5 天的数据算"5 日均线"，就是在制造虚假信号。

## 2. `.shift()` = SQL 的 `LAG()` / `LEAD()`
```python
df['prev_close'] = df['close_price'].shift(1)
```
* **`shift(1)`**：整列数据物理下移一格。第一行变成 NaN（因为"昨天"不存在），而原来第一行的值现在出现在第二行的位置。效果等同于 SQL 里的 `LAG(close_price, 1)`。
* **`shift(-1)`**：反向上移，等同于 `LEAD()`。但在量化建模中**几乎不会用**，因为这意味着你在偷窥未来数据（前视偏差）。
* **金融本质**：`shift` 的核心价值在于让"今天"和"昨天"出现在**同一行**里，使得"跨时间的比较计算"变成了简单的列运算。

## 3. 收益率的手工推导 vs 内置函数
你的代码展示了两种方式求日收益率：
```python
# 方式A：手工用 shift 推导
df['daily_return'] = (df['close_price'] - df['prev_close']) / df['prev_close']

# 方式B：Pandas 一键内置
df['daily_return_verify'] = df['close_price'].pct_change()
```
* **为什么先学手工再用内置？** 因为 `pct_change()` 本质上就是在底层帮你执行了 `shift(1)` 然后做除法。如果你不理解它的推导过程，后面遇到"对数收益率"或"多期累计收益率"时会完全无从下手。
* **断言校验的意义**：你用 `assert_series_equal` 证明了手工结果与内置函数的结果在 `1e-10` 精度下完全一致，这是 Week 1 Day 7 对账方法论的延续——用两条独立路径交叉验证同一个结论。

## 4. 金叉/死叉：你的第一个交易信号
这是整段代码中最有"量化味道"的部分。你用 `shift` 构造了一个极简的状态机：
* **今天** `SMA_5 > SMA_20`，但**昨天** `SMA_5 ≤ SMA_20` → 短线动能刚刚**向上穿越**长线趋势 → **金叉**（买入信号）
* 反之 → **死叉**（卖出信号）

这里 `shift` 扮演了关键角色：如果不知道"昨天的状态"，就无法判断"今天是否发生了穿越"。这正是为什么 `shift` 在量化因子工程中出现频率极高——它是一切"状态变化检测"的原子操作。

> ⚠️ **重要提醒**：双均线交叉策略在真实交易中胜率极低（大约 40%~45%），因为它对震荡市完全无能为力，会产生大量的虚假穿越信号。但作为理解 `rolling` + `shift` 组合拳的教学项目，它是完美的。
